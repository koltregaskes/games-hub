# Build Games Like a Solo Studio

This section is for people building games with a small team, a solo setup, or an agent-assisted workflow. It is written to help you make progress on real playable games without pretending you need a huge studio before you start.

## What this section is for

- Give you a studio-grade reference that still makes sense if you are not a professional programmer.
- Show the difference between a promising prototype and a reviewable demo.
- Help you use agents without letting the project turn into vague, untestable chaos.
- Keep public-safe craft knowledge in one place so the private studio console can build on the same foundation.

## The default solo-studio model

- Start with one visible playable slice.
- Define one clear review gate at a time.
- Prefer one game loop that already feels like a product over five disconnected features.
- Keep evidence, checklists, and handoffs alongside the work.
- Move to heavier engines only when the browser slice has already proved the idea and the browser is now the bottleneck.

## What good looks like

- The player can understand the fantasy within the first minute.
- The build has a stable title flow and a stable review route.
- Audio, UI, and presentation support the fantasy instead of apologizing for it.
- The team knows what is in scope for the next gate and what is explicitly out of scope.
- Updates are explained in plain English, not hidden in code or chat history.

## Minimum studio-standard version

- One complete loop with a start, pressure, and end state.
- One visual lane that already feels intentional.
- One review URL or build entry point that works on demand.
- One short handoff that says what changed, what was proved, and what still hurts.
- One evidence pack with screenshots or desktop review images.

## Common mistakes

- Expanding the world before proving the loop.
- Calling something "done" because the code runs once on your machine.
- Letting the agent add more systems than the current review gate actually needs.
- Treating marketing, onboarding, audio, and packaging as "later" problems.
- Building private operational knowledge in ten different places.

## Practical checklist

- Define the current demo gate in one paragraph.
- Decide what platform is proving the idea right now.
- Write the review route before adding more content.
- Capture evidence every time the build meaningfully changes.
- Record what would trigger a move to a heavier engine or desktop runtime.
