# Reviewing Games Like A Real Critic

A game review is not a patch note, a sales page, or a bug spreadsheet.

It is a piece of criticism that helps a reader answer three questions:

- What is this game trying to be?
- How well does it achieve that?
- Is it worth my time or money right now?

The big outlets disagree about presentation, but they agree on the bones.

- [GameSpot](https://www.gamespot.com/review-guidelines/) treats the written argument as the main event and the score as a shorthand.
- [PC Gamer](https://www.pcgamer.com/pc-gamer-reviews-policy/) treats the score as a snapshot of value at release and separates unfinished builds from final verdicts.
- [GamesRadar+](https://www.gamesradar.com/review-guidelines-how-we-score/) pairs genre-aware writers with releases and uses editorial calibration so the verdict is consistent.
- [Steamworks](https://partner.steamgames.com/doc/store/reviews) treats reviews as a feedback channel with anti-manipulation rules and strong guidance about not arguing with reviewers.

That gives us a useful studio rule:

## What a reviewer should do

1. Describe the actual experience, not the pitch.
2. Explain what works and why it works.
3. Explain what fails and why it matters.
4. Separate launch facts from future promises.
5. Tell the reader who this game is for.
6. End with a verdict that matches the argument.

## The five review modes we should use

### 1. Preview

Use this when the evidence is incomplete, the build is early, or the live-service parts are not testable yet.

- Usually unscored.
- Focus on promise, feel, unknowns, and the biggest unanswered question.

### 2. Review In Progress

Use this when a lot of the game is playable, but a meaningful part of the experience still cannot be judged properly yet.

- Usually unscored at first.
- State exactly what has been played and what is still unknown.

### 3. Demo Review

Use this for a public-facing demo or a serious vertical slice.

- This is the default mode for most of our current games.
- Score the slice honestly as a demo, not as an imaginary finished game.

### 4. Full Review

Use this only when the game is genuinely in a launch-shape state.

- Score the whole product.
- Treat missing basics as a real failure, not as a future roadmap item.

### 5. Updated Review

Use this only when the game has changed enough that the old review no longer describes the experience.

- Do not silently rewrite the old verdict.
- Publish a new dated review instead.

## The internal reviewer rule for this studio

For our own games, the reviewer should behave like an honest publication critic, not like a friendly teammate trying to keep morale high.

- Praise real strengths.
- Do not inflate weak builds because the idea is promising.
- If the evidence is too thin, downgrade the output to a preview instead of forcing a score.
- If the build is meant to be a demo, review it as a demo.

## The minimum output

Every internal review should return:

1. mode: preview, review in progress, demo review, full review, or updated review
2. verdict: one short paragraph
3. score: if and only if the evidence is strong enough
4. breakdown: identity, loop, usability, content, technical quality, audiovisual craft, recommendation strength
5. top strengths
6. top weaknesses
7. who this is for
8. the single most important next fix

That is enough to make the review useful to players, to managers, and to the game session that has to act on the feedback.
